# en を基準に locale を作り直す手順

この手順は、`src/uagent/locales/en/LC_MESSAGES/uag.po` を基準にして、各言語の `uag.po` を作り直すときの流れをまとめたものです。

例として `ja` を使っていますが、他言語でも同じ考え方で使えます。

## 目的

- `uag.pot` から抽出された最新のメッセージセットを基準にする
- `en.po` を完全一致の参照版として整える
- 既存の対象言語 `uag.po` は退避してから、必要な訳を戻す
- 足りない分は新規に翻訳する
- `fuzzy` / `plural` / 不整合を解消する
- 不自然な翻訳や表現ゆれを修正する
- 文章として不自然な箇所がないかを通しで確認する
- 必要なら、読みやすい自然な表現に言い換える

## 手順

### 1. `babel.cfg` を確認・整理する

まず、抽出対象を整理します。

- `src/uagent/tools` を対象外にする
- `tests` を対象外にする
- 共通メッセージだけを `uag.pot` に含める

これにより、ロケールファイルに不要な内部メッセージが混ざるのを防ぎます。

### 2. `uag.pot` を再生成する

`babel.cfg` を反映した状態で、テンプレートを再生成します。

```bash
python -m babel.messages.frontend extract -F ./babel.cfg -o src/uagent/locales/uag.pot .
```

### 3. `en.po` を作り直す

`uag.pot` を元に、`en.po` を全件 `msgid == msgstr` の参照版として作り直します。

- `msgid` の一覧を最新化する
- `fuzzy` を除去する
- `plural` を整理する
- 最終的に `msgid == msgstr` を全件満たすようにする

この `en.po` は、以後の差分確認用の基準になります。

### 4. 既存の対象言語 `uag.po` を退避する

作り直す前に、既存の対象言語訳を退避します。

```text
src/uagent/locales/<lang>/LC_MESSAGES/uag.po
→ src/uagent/locales/<lang>/LC_MESSAGES/uag.po.org1
```

これで元の訳を参照できるようにします。
退避した `.org` / `.org1` は翻訳の参考として使い、新しい `uag.po` を作るときに照合します。

### 5. 新しい `uag.po` を `uag.pot` から作る

退避済みの `uag.po.org1` を参照しながら、新しい `uag.po` を作成します。

- `uag.pot` にある最新の件数を新しい対象にする
- 旧 `uag.po.org1` と一致するものは復元する
- 旧訳がないものは新規に翻訳する
- pot に存在しない古い msgid は残さない
- `fuzzy` を消す
- `plural` を整理する

### 6. 直訳ではなく自然な表現に整える

復元後に、以下を重点的に見直します。

- 対象言語として不自然な語順が残っていないか
- 技術用語の表記が揺れていないか
- `workdir`、`stdin_loop` などの内部用語が自然か
- 長い警告文が読みやすいか
- 英語の直訳になっていないか
- 文章として不自然な箇所がないかを通しで確認する
- 必要なら、読みやすい自然な表現に言い換える
- 意味は合っていても、対象言語として不自然な表現になっていないか

- 不自然な箇所は、見つけたらまとめて一気に直す
- 文章全体を通して、自然さをそろえる

必要に応じて、用語を統一します。

例:

- `LLM provider` → `LLMプロバイダー`
- `model(deployment)` → `モデル（デプロイ名）`
- `workdir` → `作業ディレクトリ`
- `stdin_loop` → `標準入力ループ`
- `embedding` → `埋め込み`

### 7. `fuzzy` / `plural` / 不整合を確認する

編集後は、機械的な整合性を確認します。

- `fuzzy = 0` であること
- `plural` が残っていないこと
- `msgid` と `msgstr` の数が一致していること
- `msgid != msgstr` の不一致が残っていないこと
- 参照先の `msgid` と内容が食い違っていないこと
- `msgctxt` や複数行メッセージの崩れがないこと

特に、翻訳を直したあとに古い `msgstr` が残っていないかを確認します。

### 8. `.mo` を再生成する

`uag.po` を更新したら、`.mo` も再コンパイルします。

```bash
python scripts/compile_locales.py
```

または、必要に応じて対象 locale だけを再生成します。

### 9. 件数と自然さを最終確認する

最終確認では、以下をチェックします。

- `entries` が `uag.pot` と一致していること
- `fuzzy = 0`
- `plural = 0`
- `msgid != msgstr` がないこと
- 不自然な翻訳が残っていないこと
- 用語の表記ゆれがないこと
- 英語のまま残すべき語と翻訳すべき語が分かれていること

必要なら、画面表示や実際の利用場面で読み直して、自然さを微調整します。

## 実際の成果

今回の再構築では、次の状態になりました。

- `en.po` は `uag.pot` と完全一致
- `target.po` は旧訳を復元しつつ、不足分を新規翻訳
- `fuzzy` / `plural` / 不整合を解消
- その後、自然さを見て追加修正
- `target.mo` も再生成済み

## 補足

- `.org` / `.org1` などは退避用バックアップとして扱う
- `en.po` は翻訳対象ではなく、参照用の基準ファイルとして使う
- `uag.po` を直接編集する前に、必ず退避を取る
- この手順は `ja` 以外の言語にもそのまま適用できる

- `#!/usr/bin/env bash`, `#`, `<#`, `#>`, `@echo off` のような shebang や記号は翻訳しない
- `OpenRouter`, `Vertex AI`, `Grok`, `Ollama`, `NVIDIA` などの製品名・識別子は翻訳しない
- `%(label)s`, `%(key)s=%(value)s` のようなプレースホルダ付きの文字列は、意味を保ったまま必要な部分だけ訳す
- `%(label)s` や `%(key)s=%(value)s` のようなプレースホルダ付きの文字列は、プレースホルダ部分はそのまま残しつつ、文章全体は対象言語に翻訳する
- `msgid == msgstr` が残っていても、翻訳不要な項目ならそのまま残してよい

- `':exit'`, `':help'`, `':quit'`, `':skills clear'` などのコマンド名は英語のままにする

上記の項目以外は、原則として翻訳する。

## まとめ

流れは次の通りです。

1. `babel.cfg` を整理する
2. `uag.pot` を再生成する
3. `en.po` を参照版として整える
4. 対象言語の `uag.po` を退避する
5. 退避版を参照して `uag.po` を作り直す
6. 不足分を新規翻訳する
7. 自然な表現に調整する
8. `fuzzy` / `plural` / 不整合を確認する
9. `.mo` を再生成して確認する
